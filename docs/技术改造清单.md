# 商业化技术改造清单（基于当前项目）

## 1. 现状评估
- 当前是纯前端静态应用（`index.html + app.js + localStorage`）
- 优点：快、轻、低成本
- 商业化缺口：
  - 无账号体系
  - 无云端数据与多端同步
  - 无支付与订单能力
  - 无权限控制（免费/会员）
  - 无服务端埋点与看板

## 2. 推荐技术路线（建议）
- 前端：维持当前 Web 前端，逐步模块化
- 后端：Node.js + NestJS
- 数据库：PostgreSQL
- 缓存：Redis
- 对象存储：阿里云 OSS 或腾讯云 COS（用于题库文件、导出文件）
- 部署：Docker + 云主机（先单区）
- 监控：Sentry + Prometheus/Grafana（至少先上错误监控）

## 3. 服务拆分（先单体后拆分）
- `auth-service`：登录、token、设备管理
- `bank-service`：题库、章节、题目版本
- `practice-service`：练习记录、考试记录、错题本
- `billing-service`：订单、支付回调、会员状态
- `analytics-service`：事件采集、指标聚合

## 4. 数据库核心表（MVP）
- `users`
  - id, mobile, nickname, created_at
- `memberships`
  - user_id, plan_type, status, started_at, expired_at
- `orders`
  - order_no, user_id, plan_type, amount, status, paid_at, channel
- `question_banks`
  - id, name, subject, version, status, created_at
- `questions`
  - id, bank_id, chapter, type, stem, options_json, answer_json, explanation
- `practice_records`
  - user_id, question_id, selected_json, is_correct, created_at
- `exam_sessions`
  - id, user_id, total, correct, duration_sec, submitted_at
- `wrong_book`
  - user_id, question_id, wrong_count, last_wrong_at
- `event_logs`
  - user_id, event_name, event_time, event_props_json

## 5. API 清单（第一期）
- 认证
  - `POST /v1/auth/login`
  - `POST /v1/auth/refresh`
- 题库
  - `GET /v1/banks/active`
  - `GET /v1/questions?chapter=&type=&page=`
- 练习
  - `POST /v1/practice/submit`
  - `GET /v1/practice/wrong-book`
- 模考
  - `POST /v1/exams/start`
  - `POST /v1/exams/:id/submit`
  - `GET /v1/exams/:id/result`
- 会员
  - `GET /v1/membership/me`
  - `POST /v1/orders/create`
  - `POST /v1/payments/callback`
- 埋点
  - `POST /v1/events/batch`

## 7. 迁移策略（避免一次性重写）
- 第 1 阶段：接入账号 + 云同步，保留本地题库
- 第 2 阶段：题库改为服务端下发，客户端仅缓存
- 第 3 阶段：接入支付和会员权益
- 第 4 阶段：接入后台运营系统与风控

## 8. 风险与应对
- 支付回调异常
  - 应对：幂等校验 + 订单状态机
- 题库版权风险
  - 应对：来源登记、授权存档、水印追踪
- 数据一致性
  - 应对：事件日志 + 对账任务 + 监控告警

## 9. 开发排期建议（工程）
- 周 1-2：后端脚手架、用户/题库/练习基础 API
- 周 3-4：会员与支付闭环、前端权益门控
- 周 5-6：运营后台最小可用版
- 周 7+：性能优化、风控、数据看板
